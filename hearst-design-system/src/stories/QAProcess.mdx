## Jira: Token Lifecycle Tracking

Every token change -- whether requested by a designer or initiated by a developer -- is tracked as a Jira ticket. This gives both roles a shared view of what's been requested, what's in progress, and what's available.

### Jira Statuses

A token ticket moves through six statuses. Each status has a clear owner and exit criteria.

### Status Flow

```
Designer                Developer              QA                    Designer
────────                ─────────              ──                    ────────

Requested ──────────► In Progress ──────► In Review ──────► Merged
(or self-serve          (branch — by          (PR open)        (main)
 via Cursor)             dev or designer)
                                                                │
                                                          push-figma
                                                          push-pencil
                                                                │
                                                                ▼
                                                            Synced ──────► Done
                                                          (Figma+Pencil)   (verified)
```

### Linking Jira to GitHub

When the developer opens a PR, they should:

1. Include the Jira ticket ID in the PR title (e.g., `DS-142: Add tooltip background tokens`)
2. Add the Jira ticket link in the PR description
3. If using the GitHub + Jira integration app, the ticket auto-transitions to **In Review** when the PR is opened and to **Merged** when it's merged

## Checkpoint 1: PR Review

When a PR touches files in `tokens/`, QA reviews it before merge. Here is the checklist.

### Naming Review

| Check | How to verify | Pass criteria |
|-------|---------------|---------------|
| Follows naming convention | Read the token key in the JSON | Matches `layer-category-element-variant-state` pattern |
| Uses existing segment values | Compare against Token Naming Guide | Uses established sizes, states, and variants |
| No typos | Visual inspection | Spelling is correct (check the Known Typos list) |
| No duplicates | Search the JSON file for similar names | No existing token with the same purpose |
| Correct prefix | Check if component-scoped | Component tokens start with `component-` |

### Value Review

| Check | How to verify | Pass criteria |
|-------|---------------|---------------|
| Color values are valid hex | Inspect the value | 6-digit hex with `#` prefix |
| Numbers are reasonable | Compare to existing scale | Spacing, radius, and sizes fit the established scale |
| Font names are correct | Cross-reference with `_meta.json` | Matches a font available in the system |
| Brand-specific values differ | Compare across brand files | Each brand has an appropriate override |

### File Placement Review

| Check | How to verify | Pass criteria |
|-------|---------------|---------------|
| Global tokens in `core/global.json` | Check file path | Tokens that are the same across all brands |
| Brand tokens in `brands/{slug}.json` | Check file path | Tokens that differ per brand |
| Semantic aliases in `semantic/aliases.json` | Check file path | Aliases that reference `$brand-N` |
| Font overrides in `_meta.json` | Check file path | Headline/secondary font overrides |

### Visual Review

| Check | How to verify | Pass criteria |
|-------|---------------|---------------|
| Component renders correctly | Run `npm run dev` or `npm run storybook` | No visual breaks |
| Works across 3+ brands | Switch brands in Storybook toolbar | Colors, fonts, and spacing adapt correctly |
| No regressions | Compare existing components | Other components still look correct |
| Build passes | Run `npm run build-tokens` | No errors in terminal |
| Validation passes | Run `npm run tokens:validate` | No reference errors |

## Checkpoint 2: Post-Sync Verification

After the PR is merged and the developer runs `push-figma` + `push-pencil`, QA (or the designer) verifies the tokens arrived correctly.

### Figma Verification

| Check | How to verify | Pass criteria |
|-------|---------------|---------------|
| Token exists | Open Variables panel, search by canonical name | Token appears in the "Hearst Design System" collection |
| Correct value per brand | Switch between brand modes | Each mode shows the expected value |
| codeSyntax.WEB is set | Inspect the variable | Shows the corresponding `--token-name` CSS variable |

### Pencil Verification

| Check | How to verify | Pass criteria |
|-------|---------------|---------------|
| Variable exists | Search variables in Pencil or use MCP `get_variables` | Variable appears with the canonical name |
| Correct value per theme | Switch brand themes in Pencil | Each theme resolves to the expected value |
| No stale duplicates | Check variable count | No duplicate entries for the same brand |

### Production Verification

| Check | How to verify | Pass criteria |
|-------|---------------|---------------|
| Storybook is deployed | Visit the production Storybook URL | New component/changes are visible |
| Component renders correctly | Switch brands in the toolbar | All brands render as expected |
| Token values match Git | Compare Storybook output to JSON values | CSS variables resolve to the correct values |
