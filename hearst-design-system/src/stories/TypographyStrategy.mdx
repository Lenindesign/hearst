# Typography Token Strategy

A proposal for simplifying the Hearst typography token architecture. This document compares our current approach with industry best practices and recommends a path forward.

## The Problem

Our current typography tokens are **deeply nested, hard to memorize, and difficult for designers to adopt**. A single headline token looks like this:

<div style={{ background: '#1e293b', borderRadius: '12px', padding: '24px', margin: '24px 0', overflowX: 'auto' }}>
  <code style={{ fontSize: '14px', color: '#f87171', fontFamily: 'monospace' }}>
    typography.component.card.core.content.title.default.5xl
  </code>
  <div style={{ fontSize: '12px', color: '#64748b', marginTop: '8px' }}>9 segments. Requires knowledge of component hierarchy to choose the right token.</div>
</div>

## How Responsiveness Works (Without Breakpoints in Names)

Instead of creating 4 tokens per element, we define **one token** and give it different values per breakpoint using **modes**.

## Component Mapping

When a component needs a specific typography style, it **aliases** a semantic role rather than defining its own deep path. This keeps the system flat while still allowing component-level documentation.
