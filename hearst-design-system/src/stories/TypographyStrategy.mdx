# Typography Token Strategy

A proposal for simplifying the Hearst typography token architecture. This document compares our current approach with industry best practices and recommends a path forward.

## The Problem

Our current typography tokens are **deeply nested, hard to memorize, and difficult for designers to adopt**. A single headline token looks like this:

<div style={{ background: '#1e293b', borderRadius: '12px', padding: '24px', margin: '24px 0', overflowX: 'auto', color: '#94a3b8' }}>
  <div style={{ fontSize: '14px', fontFamily: 'ui-monospace, SFMono-Regular, Menlo, monospace', color: '#f87171', marginBottom: '8px' }}>typography.component.card.core.content.title.default.5xl</div>
  <div style={{ fontSize: '12px', color: '#94a3b8' }}>8 dot-separated segments. Requires knowledge of the component hierarchy to choose the right token.</div>
</div>

### What we already have right

Before proposing changes, it's important to acknowledge that our token architecture already has the right **structure**:

<div style={{ display: 'grid', gap: '12px', gridTemplateColumns: 'repeat(auto-fit, minmax(280px, 1fr))', margin: '16px 0 32px' }}>
  <div style={{ border: '1px solid #bbf7d0', borderRadius: '12px', padding: '20px', background: '#f0fdf4' }}>
    <div style={{ fontWeight: 700, fontSize: '14px', color: '#166534', marginBottom: '6px' }}>3-Layer Architecture</div>
    <p style={{ margin: 0, fontSize: '13px', color: '#14532d' }}>Core primitives (font.json) → Semantic scale (scale.json size composites) → Component aliases. This layering is correct and matches industry best practice.</p>
  </div>
  <div style={{ border: '1px solid #bbf7d0', borderRadius: '12px', padding: '20px', background: '#f0fdf4' }}>
    <div style={{ fontWeight: 700, fontSize: '14px', color: '#166534', marginBottom: '6px' }}>Clean Primitive Scale</div>
    <p style={{ margin: 0, fontSize: '13px', color: '#14532d' }}>The font.size scale (4xs → 15xl) is well-defined, consistent, and already consumed via --text-token-* CSS variables in globals.css.</p>
  </div>
  <div style={{ border: '1px solid #bbf7d0', borderRadius: '12px', padding: '20px', background: '#f0fdf4' }}>
    <div style={{ fontWeight: 700, fontSize: '14px', color: '#166534', marginBottom: '6px' }}>Weight Composites</div>
    <p style={{ margin: 0, fontSize: '13px', color: '#14532d' }}>typography.size.md.bold bundles fontSize + fontFamily + lineHeight + letterSpacing + fontWeight into one token. This composite pattern is solid.</p>
  </div>
</div>

**The problem is not the architecture — it's the component layer.** The `typography.component.*` tree is over-specified, creating hundreds of tokens that are mostly 1:1 aliases of the size composites.

### Redundancy analysis

Most component tokens are **pass-through aliases**. For example, `typography.component.card.core.content.title.default.5xl` resolves to `{typography.size.5xl.bold}` — it adds 4 extra segments of nesting but no unique value. The card title alone has **20 size variants** (2xs through 15xl) in both `default` and `brand` flavors, producing 40 tokens for a single element.

**Key pattern:** Every major system converges on **semantic roles** (display, headline, body, label) with **3–5 segments** and **12–20 total tokens**. None of them encode component names or breakpoints in the token path.

Material Design 3 defines exactly 15 tokens: 5 roles (Display, Headline, Title, Body, Label) x 3 sizes (Small, Medium, Large). Each token carries responsive values per platform, not per name.

## The Recommendation: Flatten the Component Layer

We don't need to rebuild the architecture — it's already correct. We need to **replace the 189 component-layer tokens with ~15 semantic role tokens** that sit between the existing size composites and the components.

## How Responsiveness Works (Without Breakpoints in Names)

Instead of creating separate tokens per breakpoint (and doing so inconsistently), we define **one token** and give it different values per breakpoint using **modes**.

## Risks and Open Questions

<div style={{ display: 'grid', gap: '16px', gridTemplateColumns: 'repeat(auto-fit, minmax(280px, 1fr))', margin: '16px 0 32px' }}>
  <div style={{ border: '1px solid #fde68a', borderRadius: '12px', padding: '20px', background: '#fffbeb' }}>
    <div style={{ fontWeight: 700, fontSize: '14px', color: '#92400e', marginBottom: '6px' }}>Naming overlap</div>
    <p style={{ margin: 0, fontSize: '13px', color: '#78350f' }}>The proposed heading.2xl uses "2xl" as a role size, while the existing scale uses "2xl" as a primitive step (24px). These mean different things. We need clear documentation or a different naming convention (e.g., heading.xxl) to avoid confusion.</p>
  </div>
  <div style={{ border: '1px solid #fde68a', borderRadius: '12px', padding: '20px', background: '#fffbeb' }}>
    <div style={{ fontWeight: 700, fontSize: '14px', color: '#92400e', marginBottom: '6px' }}>Badge tokens are different</div>
    <p style={{ margin: 0, fontSize: '13px', color: '#78350f' }}>Badge typography tokens (24 of them) include textCase and custom composites — they're not simple aliases. These may need to remain as component-specific tokens or be handled separately.</p>
  </div>
  <div style={{ border: '1px solid #fde68a', borderRadius: '12px', padding: '20px', background: '#fffbeb' }}>
    <div style={{ fontWeight: 700, fontSize: '14px', color: '#92400e', marginBottom: '6px' }}>Figma variable mode limits</div>
    <p style={{ margin: 0, fontSize: '13px', color: '#78350f' }}>Figma's free plan limits variable modes. If we add 4 responsive modes per token, we need to verify our Figma plan supports this across all brand files.</p>
  </div>
  <div style={{ border: '1px solid #fde68a', borderRadius: '12px', padding: '20px', background: '#fffbeb' }}>
    <div style={{ fontWeight: 700, fontSize: '14px', color: '#92400e', marginBottom: '6px' }}>Brand vs. default variants</div>
    <p style={{ margin: 0, fontSize: '13px', color: '#78350f' }}>Card titles currently have both "default" and "brand" variants (same values but different font families in some brands). The role tokens need a strategy for brand font switching — likely via the existing --font-headline variable rather than duplicate tokens.</p>
  </div>
</div>
