## Flow 1: Designer Requests a Token

<div style={{ background: '#f8fafc', border: '1px solid #e2e8f0', borderRadius: '12px', padding: '24px', margin: '24px 0' }}>

### Step 1 -- Designer identifies the need

During design work in Figma or Pencil, the designer realizes a token is missing. For example: "I need a background color for the new Tooltip component."

### Step 2 -- Designer files a request

The designer creates a ticket (Jira, GitHub Issue, or Slack message) with:

<div style={{ background: '#fff', border: '1px solid #e2e8f0', borderRadius: '8px', padding: '20px', margin: '16px 0', fontFamily: 'monospace', fontSize: '13px', lineHeight: '1.8', color: '#334155' }}>
  <div style={{ fontWeight: 700, marginBottom: '8px', fontFamily: 'sans-serif', fontSize: '14px' }}>Token Request Template</div>
  <div><strong>What:</strong> Background color for Tooltip component</div>
  <div><strong>Purpose:</strong> Fill color behind tooltip text</div>
  <div><strong>Value:</strong> #1c1c1c (dark, near-black)</div>
  <div><strong>Brands:</strong> Same for all brands</div>
  <div><strong>Component:</strong> Tooltip</div>
  <div><strong>Variants:</strong> default, informational (#0d58c9), warning (#d06d16), error (#cc2828)</div>
</div>

The more detail the designer provides, the faster the developer can create the token with the correct name.

### Step 3 -- Developer creates the token

The developer opens the JSON file in VS Code / Cursor and adds:

```json
{
  "component-tooltip-background-default": {
    "type": "color",
    "value": "#1c1c1c"
  },
  "component-tooltip-background-informational": {
    "type": "color",
    "value": "#0d58c9"
  }
}
```

### Step 4 -- Developer builds and validates

```bash
npm run build-tokens        # Generates brands.ts + tokens.css
npm run tokens:validate     # Checks references are valid
npm run dev                 # Preview in browser
```

### Step 5 -- Developer opens a PR

The PR includes the JSON changes and the regenerated `brands.ts` + `tokens.css`. The PR description lists the new tokens by canonical name.

### Step 6 -- After merge, developer syncs to design tools

```bash
npm run push-figma          # Tokens appear in Figma variables
npm run push-pencil         # Tokens appear in Pencil variables
```

### Step 7 -- Developer notifies the designer

<div style={{ background: '#fff', border: '1px solid #e2e8f0', borderRadius: '8px', padding: '20px', margin: '16px 0', fontFamily: 'monospace', fontSize: '13px', lineHeight: '1.8', color: '#334155' }}>
  <div style={{ fontWeight: 700, marginBottom: '8px', fontFamily: 'sans-serif', fontSize: '14px' }}>Delivery Message Template</div>
  <div><strong>Status:</strong> Merged and synced</div>
  <div><strong>New tokens available:</strong></div>
  <div style={{ paddingLeft: '16px' }}>component-tooltip-background-default</div>
  <div style={{ paddingLeft: '16px' }}>component-tooltip-background-informational</div>
  <div style={{ paddingLeft: '16px' }}>component-tooltip-background-warning</div>
  <div style={{ paddingLeft: '16px' }}>component-tooltip-background-error</div>
  <div style={{ marginTop: '8px' }}><strong>Figma:</strong> Search for "component-tooltip" in the variables panel</div>
  <div><strong>Pencil:</strong> Use $component-tooltip-background-default</div>
</div>

### Step 8 -- Designer verifies

The designer opens Figma or Pencil, searches for the token name, and confirms it's available with the correct value.

</div>

## Flow 2: Developer Creates Tokens Proactively

<div style={{ background: '#f8fafc', border: '1px solid #e2e8f0', borderRadius: '12px', padding: '24px', margin: '24px 0' }}>

Sometimes developers create tokens while building a component, without a prior designer request.

### Step 1 -- Developer identifies the need

While implementing a component, the developer realizes new tokens are needed for theming.

### Step 2 -- Developer names the tokens

Following the naming convention from the Token Naming Guide:

```
layer - category - element - property - variant - state
```

### Step 3 -- Developer creates, builds, and merges

Same as Flow 1, Steps 3-6: edit JSON, build, validate, PR, merge, push to Figma and Pencil.

### Step 4 -- Developer notifies designers

The developer shares the new tokens so designers can start using them in future specs. The notification should include:

- The canonical names
- What each token controls
- Whether they change per brand or are global
- A Storybook link to see the component in action

</div>

## Token Status Tracking

Both designers and developers need to know the status of a token at any point. Here's how to track it using existing tools.

### Status Definitions

| Status | Meaning | Who sets it |
|--------|---------|-------------|
| **Requested** | Designer has asked for a new token | Designer (via ticket) |
| **In Progress** | Developer is creating the token in a branch | Developer (PR opened) |
| **In Review** | Token is in a pull request awaiting approval | Developer (PR ready) |
| **Merged** | Token is in the `main` branch | Developer (PR merged) |
| **Synced** | Token is available in Figma and Pencil | Developer (after push-figma + push-pencil) |
| **Deployed** | Token is live on the production site | Automatic (Netlify) |

## The Complete Pipeline

From request to production, every token passes through these stages. Each stage has a clear owner and tool.

```
DESIGNER                          DEVELOPER                         DESIGNER
────────                          ─────────                         ────────

1. Identify need            2. Create token in JSON          7. Verify in Figma
   (Figma / Pencil)            (VS Code / Cursor)              (Variables panel)
        │                           │
        │  Request via              │  npm run build-tokens    8. Verify in Pencil
        │  Jira / Slack             │  npm run tokens:validate    ($token-name)
        │  GitHub Issue             │
        ▼                      3. Preview locally              9. Use in designs
   Token Request                   (npm run dev)                  (Figma / Pencil)
                                    │
                               4. Open PR
                                   (Git / GitHub)
                                    │
                               5. Merge to main
                                   (GitHub)
                                    │
                               6. Sync to design tools
                                   npm run push-figma
                                   npm run push-pencil
                                    │
                                    ▼
                               Netlify auto-deploys
                               (Storybook + App)
```

## Quick Reference: Who Does What

| Action | Designer | Developer |
|--------|----------|-----------|
| Request a new token | Files a ticket with the template above | -- |
| Name the token | Suggests a purpose (developer finalizes the name) | Applies the naming convention |
| Choose the value | Provides the value (hex, px, font) | Enters it in the JSON file |
| Decide which file | -- | Chooses core, semantic, or brand JSON |
| Build and validate | -- | Runs `build-tokens` and `tokens:validate` |
| Review the PR | Optional: reviews the values | Creates and merges the PR |
| Sync to Figma | Verifies in the Variables panel | Runs `push-figma` |
| Sync to Pencil | Verifies with `$token-name` | Runs `push-pencil` |
| Use in designs | Applies the variable in Figma / Pencil | -- |
