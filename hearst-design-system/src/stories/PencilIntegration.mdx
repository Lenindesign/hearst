## What Is Pencil?

Pencil is a design tool that uses `.pen` files — structured documents containing components, layouts, and design tokens. It sits **between Figma and code** as the handoff layer.

```
Figma                    Pencil                     Code
(exploration)            (handoff)                  (production)
─────────────            ────────                   ──────────

Design freely    ───►    Rebuild approved     ───►  Implement in React
Iterate                  components with            using the same
Get approval             production tokens          token values
```

### Why Pencil exists in the workflow

| Problem | How Pencil solves it |
|---------|---------------------|
| Figma designs use approximate values | Pencil uses **exact production tokens** |
| "Figma says one thing, code does another" | Pencil components are **pixel-identical** to what ships |
| Developers can't read Figma programmatically | Pencil is readable via **MCP tools** in Cursor |
| LLMs can't inspect Figma designs | LLMs read Pencil specs via `batch_get` and `get_screenshot` |
| No single source for component specs | Pencil is the **approved spec** for every component |

## How It Connects to Git

Pencil is a **consumer** of the token system, not the source of truth. Token values flow from Git into Pencil.

```
┌─────────────────────────────────────────────────────────────────┐
│                                                                 │
│   Git  tokens/              npm run push-pencil                 │
│   ★ SOURCE OF TRUTH ★  ──────────────────────►  .pen file      │
│                                                                 │
│   core/global.json                               Variables:     │
│   brands/{slug}.json                             $brand-1       │
│   brands/_meta.json                              $space-md      │
│                                                  $content-      │
│                                                   default       │
│                                                  ... (595+      │
│                                                   per brand)    │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### The sync flow

| Step | What happens |
|------|-------------|
| 1 | Developer edits a token JSON file in `tokens/` |
| 2 | Runs `npm run build-tokens` to generate CSS + TypeScript |
| 3 | Commits and merges to `main` |
| 4 | Runs `npm run push-pencil` |
| 5 | Script reads all token files and calls `set_variables` via MCP |
| 6 | The `.pen` file updates — all components using those variables reflect new values |

## Pencil Variables

Pencil variables mirror the Git token structure. Every token in `tokens/` has a corresponding variable in the `.pen` file.

### Variable naming

Pencil uses `$` prefix for variable references:

| Git token | Pencil variable | CSS variable |
|-----------|----------------|-------------|
| `brand-1` | `$brand-1` | `--brand-primary` |
| `brand-2` | `$brand-2` | `--brand-2` |
| `palette-neutral-200` | `$palette-neutral-200` | `--palette-neutral-200` |
| `palette-content-default` | `$palette-content-default` | `--palette-content-default` |
| `palette-background-subtle-brand` | `$palette-background-subtle-brand` | `--palette-background-subtle-brand` |
| `space-md` | `$space-md` | `--space-md` |
| `space-xl` | `$space-xl` | `--space-xl` |
| `border-radius-2xs` | `$border-radius-2xs` | `--border-radius-2xs` |

### Variable categories in Pencil

| Category | Count | Examples |
|----------|-------|---------|
| **Brand colors** | 14 | `$brand-1` through `$brand-14` |
| **Palette brand** | 14 | `$palette-brand-1` through `$palette-brand-14` |
| **Palette neutral** | ~15 | `$palette-neutral-100`, `$palette-neutral-lightest` |
| **Palette alert** | ~20 | `$palette-alert-danger-100`, `$palette-alert-success-100` |
| **Content** | ~10 | `$palette-content-default`, `$palette-content-brand` |
| **Background** | ~10 | `$palette-background-subtle-brand` |
| **Spacing** | 11 | `$space-xs`, `$space-md`, `$space-2xl` |
| **Border** | ~8 | `$border-radius-2xs`, `$border-width-sm` |
| **Component** | ~200+ | `$component-button-background-primary-solid-default` |
| **Typography** | ~100+ | `$typography-card-product-content-title-lg` |

## Component Structure in Pencil

### Naming convention

Components in Pencil follow this pattern:

```
Component/{Category}/{Name}
```

**Examples:**

| Pencil name | What it is |
|-------------|-----------|
| `Component/Button/Primary` | Primary button atom |
| `Component/Card/Vertical` | Vertical card molecule |
| `Component/Badge/Primary` | Badge atom |
| `Newsletter Promo` | Newsletter signup molecule |
| `Hero Section` | Hero organism |

### Atomic design in Pencil

| Level | Examples in Pencil |
|-------|-------------------|
| **Atoms** | Button, Badge, Input, Link, Divider, Icon |
| **Molecules** | Article Card, Newsletter Promo, Right Rail Card, Trending Card |
| **Organisms** | Hero Section, Collection List, Right Rail, Footer, Navigation |
| **Templates** | Brand Home Page (full page composition) |

## MCP Tools Reference

Pencil is accessed programmatically through MCP (Model Context Protocol) tools in Cursor. These tools let developers and LLMs read specs, inspect designs, and push updates.

### Reading tools

| Tool | What it does | When to use |
|------|-------------|-------------|
| `get_editor_state` | Returns the currently open `.pen` file and selection | Start of any task — understand current context |
| `batch_get` | Search for nodes by pattern or read by ID | Reading component structure, finding elements |
| `get_screenshot` | Returns a visual screenshot of a node | Verifying designs visually, comparing to code |
| `get_variables` | Returns all variables and themes in the file | Checking current token state in Pencil |
| `snapshot_layout` | Returns computed layout rectangles | Understanding spatial relationships |
| `search_all_unique_properties` | Finds all unique property values in a tree | Auditing designs for consistency |

### Writing tools

| Tool | What it does | When to use |
|------|-------------|-------------|
| `set_variables` | Add or update variables in the file | Syncing tokens from Git (`push-pencil`) |
| `batch_design` | Insert, update, copy, replace, delete, move nodes | Creating or modifying components |
| `replace_all_matching_properties` | Bulk-replace properties across a tree | Refactoring designs, updating tokens |
| `open_document` | Open a `.pen` file or create a new one | Switching between files |
| `export_nodes` | Export nodes as PNG/JPEG/WEBP/PDF | Generating assets |

### Design guidelines tools

| Tool | What it does | When to use |
|------|-------------|-------------|
| `get_guidelines` | Returns design rules for a topic | Before designing (topics: `code`, `table`, `tailwind`, `landing-page`, `slides`, `design-system`, `mobile-app`, `web-app`) |
| `get_style_guide_tags` | Returns available style guide tags | Discovering design inspiration options |
| `get_style_guide` | Returns a style guide by tags or name | Getting design inspiration for screens/apps |
| `find_empty_space_on_canvas` | Finds open space for new content | Placing new components without overlap |

## Developer Workflow with Pencil

### Building a new component

```
1. Check Pencil for the approved design
   ┌─────────────────────────────────────────────┐
   │  MCP: batch_get("Component/Card/Vertical")  │
   │  → Returns node tree with all properties     │
   │  → Shows which variables are used            │
   │  → e.g. fill: $brand-1, padding: $space-md  │
   └─────────────────────────────────────────────┘

2. Get a visual reference
   ┌─────────────────────────────────────────────┐
   │  MCP: get_screenshot(nodeId)                 │
   │  → Returns a screenshot of the component    │
   │  → Compare against your React implementation │
   └─────────────────────────────────────────────┘

3. Map Pencil variables to CSS
   ┌─────────────────────────────────────────────┐
   │  Pencil: $brand-1                            │
   │  CSS:    var(--brand-primary)                │
   │                                              │
   │  Pencil: $space-md                           │
   │  CSS:    var(--space-md)  or  gap-4          │
   │                                              │
   │  Pencil: $palette-content-default            │
   │  CSS:    var(--palette-content-default)       │
   └─────────────────────────────────────────────┘

4. Implement in React using the same tokens
5. Verify with 3+ brands in the dev server
6. Add a Storybook story
```

### Reading component specs (example)

To inspect a component in Pencil from Cursor:

```
// Read the component structure
batch_get({ patterns: ["Component/Button/Primary"] })

// Returns something like:
{
  type: "frame",
  name: "Component/Button/Primary",
  fill: "$component-button-background-primary-solid-default",
  cornerRadius: "$border-radius-2xs",
  padding: "$space-sm $space-md",
  children: [
    {
      type: "text",
      content: "Button Label",
      fill: "$component-button-content-primary-solid-default",
      fontSize: 14,
      fontWeight: 600
    }
  ]
}
```

## LLM Workflow with Pencil

When an LLM (Cursor, Claude, etc.) needs to make design system changes:

| Step | MCP tool | Purpose |
|------|----------|---------|
| 1 | `get_editor_state` | Check which `.pen` file is open |
| 2 | `batch_get` | Read the component to understand current state |
| 3 | `get_screenshot` | Visually verify the current design |
| 4 | Edit token JSON in `tokens/` | Make the change in Git (source of truth) |
| 5 | `npm run build-tokens` | Generate CSS + TypeScript |
| 6 | `npm run push-pencil` | Sync changes back to Pencil |
| 7 | `get_screenshot` | Verify the update in Pencil |

## Designer Workflow with Pencil

### Creating a new component

1. **Design in Figma** — explore layouts, iterate, get approval
2. **Open `hearst-brands.pen`** in Pencil
3. **Rebuild the component** using `batch_design` with production tokens
4. **Use ONLY existing variables** — `$brand-1`, `$content-default`, `$space-md`, etc.
5. **Name it properly** — `Component/{Category}/{Name}`
6. **Verify** with `get_screenshot`

### Editing token values (designer self-serve)

Designers can also edit token values directly in Cursor with AI assistance:

1. Open Cursor and describe the change (e.g. "Change Cosmo's brand-1 to #cc0000")
2. The AI creates a branch, edits the JSON value, runs `build-tokens` + `tokens:check`
3. The AI opens a PR for developer review
4. After merge, developer runs `push-pencil` to sync Pencil

### Rules for Pencil components

| Do | Don't |
|----|-------|
| Use `$variable` references for all colors | Hardcode hex values like `#1B5F8A` |
| Use `$space-*` variables for all spacing | Use arbitrary pixel values |
| Follow the naming convention | Use inconsistent names |
| Keep components in sync with production | Let Pencil drift from code |

## Keeping Pencil in Sync

```
Developer edits token JSON
         │
         ▼
  npm run build-tokens
  ├── brands.ts   (React)
  └── tokens.css  (CSS)
         │
         ▼
  npm run push-pencil
         │
         ▼
  hearst-brands.pen updated
  ├── All variables reflect new values
  └── Components using those variables update automatically
```

If Pencil ever gets out of sync, running `npm run push-pencil` re-syncs all variables from the Git token files.
